# OhMyPm 追问规则

## 1. 目标

追问不是聊天补充，而是当前动作被阻断后的解除动作。

它的目标是：

- 只追当前最阻塞推进的缺口
- 让提问直接服务当前动作能否继续
- 把结果回写到项目记忆和状态文件

前置边界：

- ask-back 只服务真实阻塞
- 虚构问题、占位口径、内部矛盾不得抛给 PM

## 2. 适用场景

以下情况优先触发追问，而不是直接继续生成：

- 当前需求任务仍无法定义
- 范围边界不清
- 关键事实缺失
- 关键规则口径不清
- 关键风险未知，足以推翻当前方案
- 当前交付深度和预期不明确

至少必须覆盖以下强触发点：

- 先回应时被关键事实缺口卡住
- 开工检查前仍有待确认项
- 变更分类还未得到 PM 确认
- 范围边界未确认，但已经影响模块清单、工时或排期判断

## 3. 提问原则

- 追问必须服务当前动作，不得发散
- 优先一次只追一个问题
- 问题必须翻译成人话
- PM 回答的是业务判断，不是替系统填写内部字段
- 若已有证据可从资料中查出，优先查证，不要先问人

若同时需要确认两件事，顺序固定为：

1. 是否仍属于当前版本范围？
2. 即使仍在当前版本范围内，是否要按独立新模块处理？

## 4. 运行时触发

以下位置必须检查 ask-back：

- `scripts/tools/stage-gate.ps1`
- `omp-reply`
- `omp-ready`
- `omp-change`
- 任何准备进入正式交付的入口

检查到后必须：

1. 中断当前推进
2. 生成最小问题
3. 转入 `omp-check`
4. 等待 PM 回答后再回写状态

交互规则：

- ask-back 必须主动触发，不得等待 PM 想起命令
- `/ompcheck` 只能作为显式入口，不是 ask-back 的必要前提
- ask-back 输出最后必须收口为：
  - `现在只需要你回答的唯一问题是：...`
- 若当前问题本质上属于内部修正，则直接给内部修正方向，不转成 PM 追问

## 5. 输出要求

一次有效追问至少要给出：

- 当前为什么不能继续推进
- 当前最阻塞的问题是什么
- 回答后会解除哪一项阻塞

## 6. 回写原则

- 若用户已回答：清理对应阻塞项并更新事实
- 若仍未回答：保留到 `pending_confirmations`
- 若回答推翻当前方案：同步更新 `本轮变化点` 或进入复写判断
- 内部修正项不得写成面向 PM 的 `pending_confirmations`
