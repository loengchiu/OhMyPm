# OhMyPm 方法论摘要

说明：

- 本文件保留方法论的人读摘要
- 方法论不再作为独立运行时 contract 强制读取
- 其约束应融入固定动作流、技能说明和对应规则中执行

## 1. 对外交互硬规则

- 默认自然语言是主入口
- 命令只作为调试、强制指定和高级用户入口
- 每次对外输出最后，必须只保留一个“下一步唯一动作”
- 若当前存在待确认项或 PM 决策缺口，输出必须收口为：
  - `现在只需要你回答的唯一问题是：...`
- 若当前不存在待确认项，输出必须收口为：
  - `现在建议你做的下一步是：...`
- 一次只推进一个动作
- 若当前动作被阻断，不允许同时给多个动作菜单

## 2. 场景边界硬规则

- 必须先判断当前是 `真实项目协作` 还是 `机制验证样例 / demo`
- 若是样例 / demo：
  - 对外必须先标清“仅用于机制验证”
  - 不得要求 PM 回答样例中的业务口径
  - 样例业务缺口只能用占位值、内部修正或显式缺口标注处理
- 只有真实项目协作，才允许把业务规则缺口转成 ask-back

## 3. 分层加载硬规则

- 先判断入口，再读状态，再做决策
- 默认只读当前动作对应的一个 skill
- 默认只读当前动作必需的少量规则
- 长材料和外部知识只做局部回查
- 长文生成后只保留摘要、索引和稳定路径

禁止：

- 默认并行读多个 skill
- 默认预读多个规则
- 把整篇长材料直接压进活跃上下文

## 4. 固定分析顺序

遇到真实需求时，优先按以下顺序思考：

1. 先判当前要做哪个固定动作
2. 再判范围
3. 再判结构
4. 再判表达载体
5. 最后判估算和推进条件

遇到样例需求时，优先按以下顺序思考：

1. 先标清这是样例还是真实项目
2. 再判当前要验证的是哪条机制
3. 再决定哪些业务内容只需要占位，不需要 PM 回答
4. 最后才生成样例原型或样例 PRD

## 5. 总约束

- 不得先写产物再倒推任务
- 不得先写细节再补结构
- 不得用模板填空代替边界判断
- 不得把未确认范围包装成已确认范围
- 不得把结构性变化包装成普通补充
