# OhMyPm 分层加载规则

## 1. 目标

分层加载不是建议，而是运行时硬规则。

目标：

- 先用最小上下文判断当前动作
- 再按动作只读一个 skill
- 再按动作只读必要 contract
- 长材料和外部知识只做局部回查
- 长文生成后必须摘要回收

## 2. 五层加载结构

### 第 0 层：最小状态层

默认读取：

- `docs/ohmypm/ohmypm-status.json`
- `docs/ohmypm/ohmypm-memory.md` 的最小必要摘要

本层只允许回答：

- 当前在做什么
- 当前处于哪个阶段
- 当前是否有待确认项
- 当前是否已有稳定产物路径
- 当前属于真实项目协作还是样例/演示场景

禁止：

- 直接整篇读入长版项目记忆
- 直接读取 system memory
- 直接读取外部知识
- 直接预读多个 skill
- 直接预读多个 contract
- 未先判清真实/样例边界就把样例问题当真实问题推进

### 第 1 层：单 skill 意图层

默认行为：

- 先根据自然语言判断当前意图
- 先根据最小状态判断当前属于真实协作还是样例/演示
- 只读取当前意图对应的一个 skill

允许的默认路由：

- 初始化项目 -> `omp-intake`
- 新需求 / 先回应一下 -> `omp-respond`
- 根据反馈继续调整 -> `omp-align`
- 有阻塞问题需要 PM 回答 -> `omp-ask-back`
- 检查能否进入正式交付 -> `omp-preflight`
- 做交付型原型 -> `omp-deliver-prototype`
- 写正式 PRD -> `omp-deliver-prd`
- 开评审 / 做评审材料 -> `omp-review`
- 正式交付后处理新增变化 -> `omp-change`
- 修正已有产物 -> `omp-fix`

禁止：

- 默认同时读取多个 skill
- 让 PM 先自己判断命令
- 为了保险把整个技能目录一起读入
- 在样例/演示场景下默认向 PM 追问虚拟业务口径

### 第 2 层：单动作 contract 层

默认行为：

- 只读取当前动作必须的 contract

最小读取规则：

- 所有长文或长材料动作，补读 `contracts/context-guard.md`
- `omp-ask-back` 补读 `contracts/ask-back.md`
- `omp-preflight` 补读 `contracts/gates.md`
- `omp-deliver-prototype` / `omp-deliver-prd` 补读 `contracts/delivery.md`
- `omp-review` 补读 `contracts/review.md`
- `omp-change` 补读 `contracts/gates.md`
- 涉及记忆读写时补读 `contracts/memory.md`
- 涉及外部知识时补读 `contracts/knowledge.md`

禁止：

- 为了保险一次读很多 contract
- 未进入当前动作前预读不相关 contract

### 第 3 层：局部回查层

默认行为：

- 外部知识、系统记忆卡、长材料都只允许局部回查
- 先读摘要、索引、目录、结构化提取
- 只有摘要不足时，再回查局部原文片段

禁止：

- 整篇整包载入外部知识
- 整篇整包载入长版材料
- 一次性把多个长文全塞入当前上下文

### 第 4 层：摘要回收层

默认行为：

- 长文生成后，活跃上下文只保留：
  - 摘要
  - 索引
  - 稳定路径

保留方式：

- 已生成产物写入稳定路径
- 当前上下文只保留“这份产物解决了什么、路径在哪、还有什么缺口”

禁止：

- 长文生成后继续把全文挂在活跃上下文里
- 把历史长文不断堆回后续动作

## 3. 对外交互规则

- 对外默认表现为一个协作型大 skill
- 用户默认只用自然语言
- 系统自己判断当前动作
- 命令流只作为调试入口、强制入口、高级入口
- 若当前是样例/演示场景，必须主动标记“这是机制验证，不是向 PM 收真实业务口径”

## 4. 输出收口规则

每次输出最后只能收口为以下两类之一：

- `现在建议你做的下一步是：...`
- `现在只需要你回答的唯一问题是：...`

禁止：

- 给 PM 一串菜单自己选
- 直接把内部状态机术语丢给 PM
- 让 PM 通过 usage 或 runbook 决定下一步
