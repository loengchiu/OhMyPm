# OhMyPm 交付锚点契约

## 1. 作用

本契约只做一件事：

- 约束 `omp-prd` 在写详细需求说明时，必须按锚点生成，不能靠猜扩写

补充说明：

- 锚点是追溯元数据在 PRD 详细需求说明中的直接落点
- 锚点契约负责“怎么锚”
- 追溯元数据契约负责“为什么锚、锚到哪里、缺锚时怎么阻断”

当前详细需求说明固定结构：

1. 模块
2. 页面
3. 动作

对应固定锚点：

1. 模块锚点
2. 页面锚点
3. 动作锚点

## 2. 最小规则

- 写模块前，必须先有模块锚点
- 写页面前，必须先有模块锚点和页面锚点
- 写动作前，必须先有页面锚点和动作锚点
- 任一层锚点不足时，不得直接补正文
- 缺口只允许进入三种处理：
  - 局部回查
  - 显式标缺
  - 转待确认

## 2.1 双层锚点

交付锚点固定分为人读层和机读层。

人读层：

- PRD、原型、评审记录等对外产物只展示自然标题和页面编号
- 页面编号可使用 `P1 / P2 / P3`
- 页面内标注只显示小数字，如 `01 / 02 / 03`
- 正文标题使用中文动作名，如 `提交反馈`

机读层：

- 完整组合锚点只保存在 `.ohmypm` 内部文件
- 组合锚点格式为 `Mxx-Pxx-Axx`
- `Mxx` 表示模块，`Pxx` 表示页面，`Axx` 表示页面内动作
- 组合锚点用于 PRD、原型、评审、修复之间的一致性检查

边界：

- 人读产物不展示组合锚点
- 人读产物不展示 `anchor_id`、`rules_ref`、`prototype_ref` 等机读字段名
- PRD 标题不写组合锚点
- 原型可在 HTML 内部使用隐藏属性保存组合锚点，但页面可见区域只显示小数字

## 3. 三层锚点最少要回答什么

### 3.1 模块锚点

至少回答：

- 内部模块编号
- 模块名称
- 模块边界
- 模块涉及的主要页面或能力
- 当前版本是否处理该模块

### 3.2 页面锚点

至少回答：

- 内部页面编号
- 人读页面编号
- 页面名称
- 所属模块
- 页面承担什么事
- 页面上的关键区域或关键操作

### 3.3 动作锚点

至少回答：

- 内部动作编号
- 动作名称
- 动作发生在哪个页面
- 谁发起这个动作
- 动作带来什么状态或结果变化

## 4. 合格来源

优先使用现有稳定材料，不重新发明来源。

### 4.1 模块锚点来源

- 当前版本方案
- 功能清单
- 已确认的模块归属结论

### 4.2 页面锚点来源

- 页面结构
- 已稳定原型
- 已确认页面定义

### 4.3 动作锚点来源

- 页面关键操作
- 功能点说明
- 已有 PRD 中已确认的动作小节
- 原型中可确认的操作流

## 5. 生成顺序

详细需求说明固定按下面顺序分块：

1. 一次只写一个模块
2. 模块内一次只写一个页面
3. 页面内一次只写一个动作

默认读取边界：

- 写模块时，只读当前模块锚点
- 写页面时，只读当前模块摘要 + 当前页面锚点
- 写动作时，只读当前模块摘要 + 当前页面锚点 + 当前动作锚点

禁止：

- 为了保险整篇读完功能清单、页面结构、旧 PRD
- 写一个动作时跨很多页面取材
- 写一个模块时把其他模块正文一起塞进上下文

## 6. 缺口判定

以下情况统一视为锚点不足：

- 模块找不到明确边界
- 页面找不到明确归属
- 动作找不到明确页面落点
- 动作只有“看起来合理”，但没有来源依据

锚点不足时：

- 不得直接写成确定性正文
- 必须输出缺口说明或待确认项

## 7. 正文边界

锚点是内部生成约束，不是正文内容。

正式 PRD 正文中：

- 只写业务内容
- 页面可保留 `P1 / P2` 这类页面级编号
- 不写“模块锚点 / 页面锚点 / 动作锚点”这些术语
- 不写 `Mxx-Pxx-Axx` 这类组合锚点
- 不写 `anchor_id`、`rules_ref`、`prototype_ref` 这类机读字段名
- 不写 OMP 的方法论说明
- 不写“本段由哪个契约承接”这类元话语

## 8. 对 `omp-prd` 的强制要求

- 进入详细需求说明生成时，必须遵守本契约
- 若当前需求规模较大，必须按模块、页面、动作逐层切块
- 若动作锚点不足，不得一口气生成整章详细需求说明
- 长文生成后只保留：
  - 模块摘要
  - 页面摘要
  - 动作索引
  - 产物稳定路径
