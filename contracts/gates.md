# OhMyPm 门禁规则

## 1. 总原则

门禁锁的不是需求方，而是推进动作。

- 需求方可以继续提需求
- 但不是所有时刻都允许对外承诺、进入正式交付、把新增内容直接并入当前版本

## 2. 四层门禁

### 2.1 回应门禁

作用：判断当前是否已经能形成一版可信回应。

检查项：

- 当前需求任务明确
- 至少有一版当前版本方案
- 能说清想解决什么问题
- 能说清准备怎么做
- 能给出粗粒度工时或量级判断
- 未确认事实和未澄清问题已显式记录

不过的处理：

- 继续内部整理
- 不对外给出像承诺一样的正式回应

### 2.2 对齐推进门禁

作用：判断当前是否值得进入下一轮对齐，而不是低质量散聊。

检查项：

- 本轮变化点明确
- 新增内容已归并到模块或事实状态
- 当前建议明确
- 已能判断新增内容是否影响主结构

不过的处理：

- 先内部整理
- 暂不推进下一轮正式对齐

### 2.3 正式交付门禁

作用：判断能否从回应/校验循环进入正式交付。

前置条件：

- 需求方认可当前版本方案
- 没有新增会改变主结构的补充
- 当前方案方向无明显争议
- 模块、工时、排期影响已有稳定判断

交付前检查至少包含：

- 目标闭合
- 主流程闭合
- 模块闭合
- 关键未澄清项不会推翻方案
- 预估可讲清
- 正式表达路径明确

不过的处理：

- 回退到回应/校验循环
- 不开始交付型原型
- 不开始正式 PRD

### 2.4 变更门禁

作用：控制正式交付后新增内容的进入方式。

检查项：

- 新增内容属于哪一类：
  - 轻微补充
  - 模块内补充
  - 新模块
  - 主结构变化
- 是否影响：
  - 模块清单
  - 方案预估工时
  - 排期影响
  - 主流程
  - PRD 范围定义

处理规则：

- 轻微补充可并入当前交付，但要记录变化
- 新模块或主结构变化不得默认吞入，必须重开对齐或转为变更
