# OhMyPm 门禁规则

## 1. 总原则

门禁锁的不是需求方，而是推进动作。

- 需求方可以继续提需求
- 但不是所有时刻都允许对外承诺、进入正式交付、把新增内容直接并入当前版本

## 2. 四层门禁

### 2.1 回应门禁

作用：判断当前是否已经能形成一版可信回应。

检查项：

- 当前需求任务明确
- 至少有一版当前版本方案
- 能说清想解决什么问题
- 能说清准备怎么做
- 能给出粗粒度工时或量级判断
- 未确认事实和未澄清问题已显式记录
- 最小全局追溯元数据已成形

不过的处理：

- 继续内部整理
- 不对外给出像承诺一样的正式回应

建议回退类型：

- `internal_repair`
- `need_materials`

### 2.2 对齐推进门禁

作用：判断当前是否值得进入下一轮对齐，而不是低质量散聊。

检查项：

- 本轮变化点明确
- 新增内容已归并到模块或事实状态
- 当前建议明确
- 当前已能初判本次是 `iteration / new_build / hybrid` 之一，或已明确为什么还不能判断
- 已能判断新增内容是否影响主结构
- 至少能看出当前模块锚点

不过的处理：

- 先内部整理
- 暂不推进下一轮正式对齐

建议回退类型：

- `internal_repair`

### 2.3 正式交付门禁

作用：判断能否从回应/校验循环进入最小交付主链。

前置条件：

- 需求方认可当前版本方案
- 没有新增会改变主结构的补充
- 当前方案方向无明显争议
- 模块、工时、排期影响已有稳定判断
- 建设类型已稳定判断，不再处于“暂不能判断”

开工检查只保留以下六项高价值检查：

- 目标闭合
- 主流程闭合
- 模块闭合
- 关键未澄清项不会推翻方案
- 预估可讲清
- 正式表达路径明确

补充检查：

- 当前追溯元数据已经覆盖：版本、范围、模块、未确认项、承接边界
- 至少已经形成模块锚点；若要进入正式交付，页面和流程锚点不能完全缺失
- 若判为 `iteration`，必须已说清挂载点、改造入口和兼容约束
- 若判为 `new_build`，必须已说清新建范围、主流程和角色边界
- 若判为 `hybrid`，必须已说清存量承接部分与新建部分的边界

当前版本补充说明：

- “正式表达路径明确”优先指原型模板和 PRD 模板都已有承接位置
- 若当前只够产出说明稿，不够产出 HTML 原型本体或 PRD 骨架，则视为不过
- 若当前仍说不清页面覆盖、流程覆盖或原型/PRD 分工，也视为不过

不过的处理：

- 回退到回应/校验循环
- 不开始交付型原型
- 不开始正式 PRD

回退分流：

- 方案逻辑不闭合、模块没拆清、表达路径不清：`internal_repair`
- 关键截图、系统资料、关键人反馈缺失：`need_materials`
- 需求方推翻当前版本方案或新增内容改变主结构：`reopen_alignment`

### 2.4 变更门禁

作用：控制正式交付后新增内容的进入方式。

检查项：

- 新增内容属于哪一类：
  - 轻微补充
  - 模块内补充
  - 新模块
  - 主结构变化
- 是否影响：
  - 模块清单
  - 方案预估工时
  - 排期影响
  - 主流程
  - PRD 范围定义

处理规则：

- 轻微补充可并入当前交付，但要记录变化
- 新模块或主结构变化不得默认吞入，必须重开对齐或转为变更

分类判定权：

- AI 先初判
- PM 最终确认

## 3. 运行时枚举

为避免脚本、JSON 和宿主环境中的编码歧义，运行时状态建议固定使用以下稳定值。

### 3.1 轮次结果

`alignment_state.round_result` 只允许：

- `continue_alignment`
- `need_materials`
- `need_internal_repair`
- `ready_for_preflight`

说明：

- `reopen_alignment` 不是轮次结果
- 它只能作为 `fallback_state.fallback_type`

### 3.2 回退类型

`fallback_state.fallback_type` 只允许：

- `internal_repair`
- `need_materials`
- `reopen_alignment`

补充规则：

- `internal_repair` 和 `need_materials` 是轮次内回退，不产生新轮次编号
- 只有 `reopen_alignment` 并重新进入下一轮正式对齐时，才递增轮次编号

### 3.3 变更分类

`change_state.change_category` 只允许：

- `minor_patch`
- `within_module`
- `new_module`
- `structural_change`

补充规则：

- `new_module` 和 `structural_change` 必须显式记录 PM 确认结果

## 4. 强制规则

- 门禁不过时，不得用“先出一版再说”伪装推进
- `context-lint`、`trace-lint`、`review-pack` 是硬门禁输入，不是建议检查
- `context-lint` 失败时，不得从调研进入方案
- `trace-lint` 失败时，不得从原型或 PRD 进入下一阶段，也不得给评审通过结论
- 评审前必须生成 `review-pack.json`，评审基于冷启动评审包重新判断
- 轮次结果、回退类型、变更分类不得混用
- 任何会影响正式交付范围的新增内容，都必须先过变更门禁
- 缺少最小追溯元数据时，不得伪装推进到正式交付
- 当 `pending_confirmations` 非空时，除非当前动作明确处于 `internal_repair` 或 `need_materials`，否则不得静默推进到更重阶段
- 当 `change_state.change_category_confirmed_by_pm=false` 时，AI 只能保留初判，不得把该分类当作最终结论继续推进正式交付或正式变更处理
- 当前版本的正式交付门禁只服务最小主链，不要求同时把 review / change / fix 准备完
