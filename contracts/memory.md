# OhMyPm 记忆规则

## 1. 目标

记忆机制用于把长期有效的信息从单轮对话中剥离出来，沉淀为可持续回读的项目记忆和系统记忆。

它的目标是：

- 降低跨会话信息丢失
- 避免每次重新解释系统现状
- 为追问、评审、估算和复写提供稳定依据

## 2. 两层结构

OhMyPm 的记忆固定分两层：

- 项目记忆文件：`.ohmypm/memory.md`
- 外部系统记忆仓：`D:\work\PMsyscard\wiki\...`

## 3. 项目记忆的职责

项目记忆负责记录当前项目语义，不负责做运行时门禁真值。

至少应承载：

- 当前需求任务
- 已确认事实
- 未确认事实
- 未澄清问题
- 未澄清原因
- 当前版本方案
- 本轮变化点
- 当前建议
- 当前模块清单
- 当前方案预估工时
- 排期影响判断
- 系统记忆引用
- 新增资料记录
- 评审摘要
- 复写记录

## 4. 系统记忆卡的职责

系统记忆仓负责沉淀跨项目复用知识。

至少可承载：

- 系统定位
- 模块边界
- 页面与流程线索
- 角色与权限
- 数据与接口约束
- 存量兼容约束
- 已知风险与坑点
- 证据来源

## 5. 读取原则

- 先读项目记忆，再按需读外部系统记忆卡
- 只抽取当前动作所需的事实、规则、风险和引用
- 若项目记忆已足够，不重复载入整张系统记忆卡
- 若系统知识与当前项目冲突，必须保留冲突痕迹，不得静默覆盖

## 6. 写入原则

- 事实、推断、建议分开写
- 已确认和未确认分开写
- 当前项目结论与跨项目知识分开写
- 新结论可覆盖旧判断，但关键变更原因必须保留
- 没有证据支持的推断必须显式标注

## 7. 回写时机

以下场景默认应回写项目记忆：

- 首轮回应形成后
- 一轮对齐产生实质变化后
- preflight 形成放行或回退判断后
- review 形成统一结论后
- overwrite 或 fix 推翻上游结论后

以下场景默认应回写外部系统记忆仓：

- 发现了可跨项目复用的系统事实
- 修正了旧有系统边界认知
- 新项目证据推翻了旧卡片内容

## 8. 强制规则

- `ohmypm-status.json` 负责运行时真值，`ohmypm-memory.md` 不替代状态文件
- 不得把纯操作日志塞进项目记忆替代项目语义
- 不得把某个项目的一次性讨论细节直接写成系统记忆卡的长期事实
- OMP 只保留系统记忆引用，不在项目目录内保存系统记忆正文
- 若复写或评审推翻既有结论，记忆必须同步更新
