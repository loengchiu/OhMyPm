# OhMyPm 正式交付规则

当前版本的交付目标只固定最小主链：

- 听需求
- 回应 / 对齐
- 开工检查
- 交付型原型
- PRD

样例当前只作为模板试车台。

因此本文件当前优先回答三件事：

- 原型模板是否足够承接页面和流程
- PRD 模板是否足够承接规则和归档
- 原型与 PRD 的分工是否足够清楚
- 原型与 PRD 是否都挂在同一套追溯元数据上

补充边界：

- 正式产物不得混入 OMP 自己的方法论元话语
- 样例占位说明不得伪装成真实项目规则

## 1. 正式交付包

正式交付固定包含：

- 交付型原型
- PRD

## 2. 交付型原型

### 定位

- 当前模板试跑时的第一阅读入口
- 研发第一阅读入口

### 最低要素

- 页面落点
- 主流程
- 页面用途
- 用户动作
- 关键状态
- 关键交互标注
- 页面间关系
- 与 PRD 的引用关系

### 可复用骨架

- 原型骨架模板：`docs/templates/prototype-note.template.md`
- HTML 原型骨架模板：`docs/templates/prototype-delivery.template.html`

### 标注方式

- 使用编号标注
- 不直接把正文铺在页面上
- 点击编号查看解释

### 解释类型

- 原型解释注释
- PRD 引用注释

### 最低验收

- 研发只看原型，应能先理解 80% 的页面与流程
- PM 只看原型，应能判断模板是否已足够表达页面和流程
- 原型不替代 PRD，但必须先于 PRD 成为第一阅读入口

## 3. PRD

### 定位

- 正式归档主文件
- 原型规则补充文档

### 两层九段

1. 文档信息
2. 背景与目标
3. 本次范围
4. 整体方案概述
5. 原型与页面说明
6. 业务规则说明
7. 异常与边界说明
8. 角色、权限与数据影响
9. 验收与交付说明

### 可复用骨架

- PRD 骨架模板：`docs/templates/prd-outline.template.md`

### 原则

- 完整到足以归档
- 不把原型已表达清楚的页面内容全文重复
- 用结构完整、引用完整、规则完整保证完整性
- 当前试跑时优先验证模板是否能稳定承接，不追求一次写满所有细节

### 生成后补漏清单

PRD 初稿生成后，不得直接结束，必须做一轮补漏检查。

最少检查以下七类：

- 异常流程是否有缺口
- 边界条件是否有缺口
- 验收口径是否可执行
- 角色、权限影响是否交代
- 数据、接口影响是否交代
- 非功能要求是否需要补充
- 待确认项是否显式保留

补漏规则：

- 能直接补齐的，补进 PRD
- 不能直接补齐但会影响判断的，写入待确认项或风险项
- 不得因为补漏清单存在，就把未确认内容伪装成已确认规则
- 不得把 OMP 自己的分工说明、承接说明、模板试跑说明补进正式 PRD 正文

## 4. 分工边界

### 原型负责

- 做什么
- 在哪做
- 怎么走
- 页面间怎么串
- 关键状态和动作是什么
- 编号标注如何把页面与规则挂起来

### PRD 负责

- 业务规则
- 判定条件
- 异常分支
- 权限逻辑
- 数据口径
- 存量兼容约束
- 验收标准
- 原型编号背后的规则正文

### 判定规则

- 不看页面就难理解的内容，优先进原型
- 不写成规则就容易争议的内容，优先进 PRD
- 若内容既不能靠页面讲清，也暂时不需要归档成规则，当前版本先不纳入模板试跑范围

## 5. 继承关系

- 交付型原型与 PRD 都应继承稳定对齐包
- 交付型原型与 PRD 都应继承同一套最小追溯元数据
- 原型先承接流程、页面、状态和编号标注
- PRD 再承接规则、异常、权限、数据影响和验收
- 两者应互补，不应整段重复

## 5.1 最小追溯要求

交付前至少应具备：

- 当前版本
- 当前范围摘要
- 当前模块锚点
- 当前页面锚点
- 当前流程 / 动作锚点
- 当前未确认项
- 原型 / PRD 承接边界

强制规则：

- 原型不得脱离页面、流程、动作锚点生成
- PRD 不得脱离模块、页面、动作锚点扩写
- 原型标注编号和 PRD 规则引用必须能相互对齐
- 原型和 PRD 的正式正文都不得出现 OMP 契约术语或方法论元话语

## 6. 当前试跑口径

当前版本只要求模板能稳定跑通以下最小结果：

- preflight 后能直接产出一版 HTML 原型本体
- 原型完成后能直接产出一版 PRD 骨架正文
- PRD 完成后，已能回头判断：
  - 原型模板是否够用
  - PRD 模板是否够用
  - preflight 规则是否过松或过紧

这一步不以 review / change / fix 的体验完整度作为交付目标。
