# OhMyPm 正式交付规则

当前版本的交付目标只固定最小主链：

- 听需求
- 回应 / 对齐
- 开工检查
- 交付型原型
- PRD

因此本文件当前优先回答三件事：

- 原型模板是否足够承接页面和流程
- PRD 模板是否足够承接规则和归档
- 原型与 PRD 的分工是否足够清楚
- 原型与 PRD 是否都挂在同一套追溯元数据上

补充边界：

- 正式产物不得混入 OMP 自己的方法论元话语
- 占位说明不得伪装成真实项目规则

## 1. 正式交付包

正式交付固定包含：

- 交付型原型
- PRD

## 2. 交付型原型

### 定位

- 研发第一阅读入口
- 页面本体优先，不是方案讲解页

### 最低要素

- 页面落点
- 主流程
- 页面用途
- 用户动作
- 关键状态
- 关键交互标注
- 页面间关系
- 与 PRD 的引用关系

### 页面本体优先规则

- 原型主舞台必须先出现页面本体，而不是大段方案说明
- 页面说明只能做短说明，不得盖过页面本体
- 原型首先要让研发看懂“页面里有什么、从哪里进、点哪里、会发生什么”

### 可复用骨架

- 原型骨架模板：`docs/templates/prototype-note.template.md`
- HTML 原型骨架模板：`docs/templates/prototype-delivery.template.html`

### 标注方式

- 使用编号标注
- 不直接把正文铺在页面上
- 编号必须贴在页面元素上
- 点击编号查看解释，不使用右侧长说明栏替代标注层
- 浮窗内容应足以替代对应 PRD 阅读
- 标注必须跟随页面切换、弹窗开关和主要状态变化同步

### 最低验收

- 研发只看原型，应能先理解 80% 的页面与流程
- 原型不替代 PRD，但必须先于 PRD 成为第一阅读入口
- 若页面看起来信息很多，但仍无法回答“元素、基本流程、基本交互是什么”，则视为不过

## 3. PRD

### 定位

- 正式归档主文件
- 原型规则补充文档

### 两层九段

1. 文档信息
2. 背景与目标
3. 本次范围
4. 整体方案概述
5. 原型与页面说明
6. 业务规则说明
7. 异常与边界说明
8. 角色、权限与数据影响
9. 验收与交付说明

### 可复用骨架

- PRD 骨架模板：`docs/templates/prd-skeleton.template.md`

### 原则

- 完整到足以归档
- 不把原型已表达清楚的页面内容全文重复
- 用结构完整、引用完整、规则完整保证完整性

### 生成后补漏清单

PRD 初稿生成后，不得直接结束，必须做一轮补漏检查。

最少检查以下七类：

- 本轮改动是否已同步到关联页面、流程、规则、权限、数据影响、验收和原型标注
- 异常流程是否有缺口
- 边界条件是否有缺口
- 验收口径是否可执行
- 角色、权限影响是否交代
- 数据、接口影响是否交代
- 非功能要求是否需要补充
- 待确认项是否显式保留

## 4. 分工边界

### 原型负责

- 做什么
- 在哪做
- 怎么走
- 页面间怎么串
- 关键状态和动作是什么
- 编号标注如何把页面与规则挂起来

### PRD 负责

- 业务规则
- 判定条件
- 异常分支
- 权限逻辑
- 数据口径
- 存量兼容约束
- 验收标准
- 原型编号背后的规则正文

## 5. 继承关系

- 交付型原型与 PRD 都应继承稳定对齐包
- 交付型原型与 PRD 都应继承同一套最小追溯元数据
- 原型先承接流程、页面、状态和编号标注
- PRD 再承接规则、异常、权限、数据影响和验收
- 两者应互补，不应整段重复
