# OhMyPm 正式交付规则

- 正式产物不得混入 OMP 自己的方法论元话语
- 占位说明不得伪装成真实项目规则

## 1. 正式交付包

正式交付固定包含：

- 交付型原型
- PRD

## 2. 交付型原型

### 最低要素

- 页面落点
- 主流程
- 页面用途
- 用户动作
- 关键状态
- 关键交互标注
- 页面间关系
- 与 PRD 的引用关系

### 最低验收

- 研发只看原型，应能先理解 80% 的页面与流程
- 原型不替代 PRD，但必须先于 PRD 成为第一阅读入口
- 若页面看起来信息很多，但仍无法回答“元素、基本流程、基本交互是什么”，则视为不过

## 3. PRD

### 正式归档骨架

1. 文档信息
2. 背景与目标
3. 本次范围
4. 整体方案概述
5. 原型与页面说明
6. 业务规则说明
7. 角色与权限说明
8. 数据字典与数据影响
9. 异常与边界说明
10. 验收与交付说明

### 生成后补漏清单

PRD 初稿生成后，不得直接结束，必须做一轮补漏检查。

最少检查以下七类：

- 本轮改动是否已同步到关联页面、流程、规则、权限、数据影响、验收和原型标注
- 异常流程是否有缺口
- 边界条件是否有缺口
- 验收口径是否可执行
- 角色、权限影响是否交代
- 数据字典、字段口径、数据流向和接口影响是否交代
- 详细需求说明是否已按需给出字段分组；无分组价值的页面是否保持轻写，而不是机械补分组
- 非功能要求是否需要补充
- 待确认项是否显式保留
- 字段落地是否完整：数据字典中的业务字段是否在详细需求说明、权限矩阵、状态规则、验收标准或待确认项中有明确落点
- 字段一致性是否完整：同一字段在页面展示、表单录入、筛选条件、统计口径和导出字段中的名称、格式、枚举、校验是否一致
- 详细需求说明中的字段改动是否已同步回全局数据字典；不得只改页面正文不改汇总层
- 动作级场景是否完整：按钮、表单、列表、弹窗、异常、降级、边界是否按需补齐，不能只写主流程

## 4. 分工边界

### 原型负责

- 做什么
- 在哪做
- 怎么走
- 页面间怎么串
- 关键状态和动作是什么
- 编号标注如何把页面与规则挂起来

### PRD 负责

- 业务规则
- 判定条件
- 异常分支
- 权限逻辑
- 数据口径
- 字段定义与枚举口径
- 存量兼容约束
- 验收标准
- 原型编号背后的规则正文

## 5. 继承关系

- 交付型原型与 PRD 都应继承稳定对齐包
- 交付型原型与 PRD 都应继承同一套最小追溯元数据
- 原型先承接流程、页面、状态和编号标注
- PRD 再承接规则、异常、权限、数据字典、数据影响和验收
- 两者应互补，不应整段重复
