# OhMyPm 阶段化 Checkpoint 契约

## 1. 作用

本契约用于把门禁结果收成统一的阶段准出卡。

目的不是增加阶段，而是让每个阶段都能回答同一组问题：

- 当前输入够不够
- 当前最低输出是什么
- 什么情况下绝不能越过
- 过了以后稳定了什么
- 不过时回退到哪里

## 2. 统一卡片结构

每个阶段的 Checkpoint 至少包含五段：

### 2.1 输入条件

- 当前阶段开始前必须已有的最小输入

### 2.2 最低输出

- 当前阶段结束后必须留下的最小结果

### 2.3 禁止越过条件

- 哪些缺口存在时，不能直接进入下一更重阶段

### 2.4 稳定基线

- 通过后，哪一类结论可以进入 `baselines` 或稳定引用范围

### 2.5 回退动作

- 不通过时，应转入：
  - `ask_back`
  - `internal_repair`
  - `need_materials`
  - `reopen_alignment`

## 3. 当前最小覆盖阶段

### 3.1 听需求

- 输入条件：
  - 用户已提出明确需求或问题
- 最低输出：
  - 已建立当前需求任务
  - 已初始化最小上下文包
- 禁止越过条件：
  - 连一句人话需求摘要都没有
  - 业务环节为空
- 稳定基线：
  - `context_package`
- 回退动作：
  - `ask_back`
  - `internal_repair`

### 3.2 回应 / 对齐

- 输入条件：
  - `context_package` 已成形
  - 当前版本号、当前理解、最小追溯元数据已开始建立
- 最低输出：
  - 当前版本方案
  - 未确认项
  - 模块级粗估
  - 模块锚点
- 禁止越过条件：
  - 范围边界仍会推翻当前主判断
  - 连模块级锚点都没有
- 稳定基线：
  - `response_plan`
  - 已稳定的模块级判断
- 回退动作：
  - `ask_back`
  - `internal_repair`
  - `need_materials`

### 3.3 开工检查

- 输入条件：
  - 当前版本方案已被需求方认可
  - 页面 / 流程锚点已开始闭合
- 最低输出：
  - 目标闭合、流程闭合、模块闭合判断
  - 进入正式交付的放行或回退结论
- 禁止越过条件：
  - `pending_confirmations` 非空
  - `anchors_state.meta.can_progress=false`
  - 页面 / 流程锚点缺失
- 稳定基线：
  - 可进入交付的当前版本
- 回退动作：
  - `ask_back`
  - `internal_repair`
  - `reopen_alignment`

### 3.4 原型 / PRD

- 输入条件：
  - 已通过开工检查
  - 原型与 PRD 承接边界明确
- 最低输出：
  - 原型：页面、流程、状态、动作、编号
  - PRD：规则、异常、权限、数据影响、验收口径
- 禁止越过条件：
  - 原型或 PRD 脱离同一套追溯元数据
  - 引用编号无法对齐
- 稳定基线：
  - `prototype`
  - `prd`
- 回退动作：
  - `internal_repair`
  - `reopen_alignment`

### 3.5 评审

- 输入条件：
  - 原型和 PRD 已形成可读版本
- 最低输出：
  - 三级风险
  - 统一结论
  - 复评清单
- 禁止越过条件：
  - 阻断项未清
  - 核心锚点失配
- 稳定基线：
  - `review_pack`
- 回退动作：
  - `internal_repair`
  - `reopen_alignment`

## 4. 强制规则

- Checkpoint 不是独立大阶段，而是阶段准出表达
- 不得只说“能继续/不能继续”，却不说明稳定了什么
- 不得把回退动作写成模糊话术
