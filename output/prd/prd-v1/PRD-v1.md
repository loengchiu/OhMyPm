## 2. 背景与目标
说明：本样例用于验证 OhMyPm 的自然语言路由、门禁、preflight、交付分工、评审和状态回写机制。以下审批业务仅作为虚拟样例，不应向 PM 追问真实业务口径。

样例占位口径：
- 金额阈值占位：`>= 100000`
- 风险标签占位：`RISK_HIGH`
- 专项复核角色占位：
  - 业务线 A -> `ROLE_SPECIAL_REVIEW_A`
  - 业务线 B -> `ROLE_SPECIAL_REVIEW_B`
  - 未命名业务线 -> `ROLE_SPECIAL_REVIEW_DEFAULT`

当前需求是在现有审批能力内部补充审批路径扩展，使需求方、评审方和研发方能够清楚理解：新增路径从哪里进入、如何影响审批主链路、哪些状态和通知会变化，以及异常情况下如何回退和留痕。

本次 PRD 目标：
- 固定本轮需求仍属于现有审批能力内部增强，不单独拆出新模块
- 说明新增审批路径对主流程、关键分支、异常流程和联动结果的影响
- 为后续详细规则、权限、数据影响和验收标准提供正式归档骨架

## 3. 本次范围
本次范围包括：
- 审批入口与待办列表对新增路径的承接
- 审批详情与处理页对新增分支的展示和处理动作
- 审批流转记录对新增路径、异常动作和结果留痕的承接
- 通知触达对新增路径通知联动的承接

本次不包括：
- 新建独立模块
- 脱离现有审批能力的新系统边界
- 与当前审批路径扩展无关的其他审批能力改造

## 4. 整体方案概述
本次方案仍以现有审批能力为主体，通过在原有链路中补充新增审批路径，实现更细的结果判断、状态同步、通知联动和留痕记录。

整体方案口径：
- 模块不变
- 复杂度上升
- 因此工时和排期判断上调

对应展示物：
- 原型负责讲清页面、流程、状态和联动
- PRD 负责讲清判定条件、规则、异常、权限、数据影响和验收

## 6. 业务规则说明

### 6.1 新增审批路径进入规则
- 当审批单同时满足以下两个条件时，进入新增审批路径：
  - 申请金额大于等于样例占位阈值 `100000`
  - 当前单据带有样例占位风险标签 `RISK_HIGH` 或命中对应样例风险规则
- 若只命中其中一个条件，则仍沿用原审批路径，不进入新增分支。
- 新增审批路径的进入判断发生在详情处理页提交审批动作之前，系统应在详情页明确展示当前是否命中新增路径。
- 仅改变内部判定但不改变外部结果的分支，可沿用原流程展示；会改变状态、通知或记录结果的分支，必须按新增路径展示和留痕。
- 以上阈值与标签仅为样例占位值，用于验证机制链路，不代表真实业务口径。

### 6.2 状态同步规则
- 主状态最少包括：`待审批`、`处理中`、`专项复核中`、`已完成`、`已退回`、`已驳回`、`已中断`。
- 当命中新增审批路径并提交专项复核后，主状态从 `处理中` 更新为 `专项复核中`。
- 当专项复核通过后，主状态更新为 `已完成`；当专项复核退回或驳回时，主状态更新为对应结果状态。
- 状态变化后，以下位置必须同步更新：
  - P1 列表状态标签
  - P2 当前状态展示
  - P3 流转记录中的动作结果
  - P4 通知结果和触达状态
- 若状态变化未引起任何列表、记录或通知变化，则应视为同步规则缺失，不允许作为通过结果进入正式交付。

### 6.3 通知联动规则
- 当详情页执行“提交专项复核”动作且成功进入新增路径时，系统向当前业务线对应的样例专项复核角色发送待处理通知。
- 当专项复核完成时，系统向当前审批责任人发送结果通知。
- 当流程沿用原路径且未进入新增分支时，不新增专项复核通知，继续沿用原有提醒规则。
- 通知至少记录以下内容：
  - 触发节点
  - 接收对象
  - 触发时间
  - 触发结果
- 若通知已触发，则 P3 必须存在对应动作记录，P1/P2 必须能看到与之匹配的状态结果。

### 6.4 留痕规则
- 以下动作必须留痕：
  - 进入详情页并命中新增路径判断
  - 提交专项复核
  - 专项复核通过
  - 驳回
  - 退回
  - 重新提交
  - 流程中断
- 留痕记录至少包含：动作名称、执行人、执行时间、执行结果、对应状态。
- 当进入新增审批路径时，轨迹中必须额外标记“新增路径已生效”，用于与原路径区分。
- 若异常动作发生，则异常留痕必须先于最终状态展示可见。

## 7. 异常与边界说明

### 7.1 异常处理范围
- 驳回
- 退回
- 重新提交
- 流程中断

### 7.2 边界说明
- 本轮新增内容仍属于现有审批能力内部增强
- 不形成新的独立外部能力边界
- 与当前路径扩展无关的改造不纳入本次范围

### 7.3 后续补充项
- 当前版本已补齐最小异常处理口径，后续仅补更细的节点差异说明

### 7.4 异常处理规则
- 驳回：当前审批责任人选择驳回后，状态更新为 `已驳回`，P3 记录驳回动作，P4 按需发送结果通知。
- 退回：当前审批责任人选择退回后，状态更新为 `已退回`，流程回到上一处理位置或上一提交人，P3 记录退回动作。
- 重新提交：发起人或指定处理角色重新提交后，系统重新执行新增路径判断，并在 P3 中追加“重新提交”记录。
- 流程中断：当关键校验失败或人工终止流程时，状态更新为 `已中断`，P3 必须显示中断原因，P4 不再继续发后续待处理通知。

## 8. 角色、权限与数据影响

### 8.1 角色影响
- 当前审批责任人
- 新增路径涉及的样例专项复核角色
- 通知接收角色

### 8.2 权限影响
- 当前审批责任人可查看审批详情、执行通过/驳回/退回动作。
- 样例专项复核角色仅在命中新增路径后获得处理权限，可执行复核通过或退回动作。
- 发起人可查看流程结果，并在允许场景下执行重新提交。
- 未在当前节点上的角色只能查看，不可执行处理动作。

### 8.3 数据影响
- 状态字段变化
- 流转记录变化
- 通知记录变化
- 查询结果和统计口径可能受影响

### 8.4 最小角色动作矩阵
| 角色 | 可查看 | 可处理 | 典型动作 |
| --- | --- | --- | --- |
| 当前审批责任人 | 是 | 是 | 通过、驳回、退回、提交专项复核 |
| 样例专项复核角色 | 命中新路径后是 | 命中新路径后是 | 复核通过、复核退回 |
| 发起人 | 是 | 条件下是 | 重新提交 |
| 其他观察角色 | 是 | 否 | 仅查看 |

### 8.5 样例角色映射
| 业务线 | 样例专项复核角色 |
| --- | --- |
| 业务线 A | `ROLE_SPECIAL_REVIEW_A` |
| 业务线 B | `ROLE_SPECIAL_REVIEW_B` |
| 其他 / 默认 | `ROLE_SPECIAL_REVIEW_DEFAULT` |

## 9. 验收与交付说明

### 9.1 主流程验收
- 验收项 A1：当同时命中样例占位阈值 `100000` 和样例占位风险标签 `RISK_HIGH` 时，详情页显示将进入新增审批路径。
- 验收项 A2：执行“提交专项复核”后，状态更新为 `专项复核中`。
- 验收项 A3：P1、P2、P3、P4 四处对该状态变化展示一致。
- 验收项 A4：专项复核通过后，主状态更新为 `已完成`。

### 9.2 分支流程验收
- 验收项 B1：只命中一个条件时，不进入新增路径，继续沿用原路径。
- 验收项 B2：命中两个条件时，进入新增路径，并在 P3 标记“新增路径已生效”。
- 验收项 B3：原路径单据不出现专项复核通知。

### 9.3 异常流程验收
- 验收项 C1：驳回后状态变为 `已驳回`，且 P3 可见驳回记录。
- 验收项 C2：退回后状态变为 `已退回`，且流程返回正确位置。
- 验收项 C3：重新提交后，系统重新执行新增路径判断。
- 验收项 C4：流程中断后，不再继续发送后续待处理通知。

### 9.4 联动影响验收
- 验收项 D1：状态变化后，P1 列表标签同步更新。
- 验收项 D2：状态变化后，P3 动作结果记录同步更新。
- 验收项 D3：触发通知的动作在 P4 可见对应结果。
- 验收项 D4：P1、P3、P4 中的结果不允许互相冲突。

### 9.5 兼容性验收
- 验收项 E1：未命中新路径的单据仍按原路径处理。
- 验收项 E2：未命中新路径的单据不会收到专项复核通知。
- 验收项 E3：原有规则在未被本次修改的范围内保持有效。
