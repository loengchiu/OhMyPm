# OhMyPm 工作流

## 1. 运行形态

OhMyPm 运行在 AI IDE / AI CLI 中。  
宿主模型通过读取项目目录内的规则、状态和记忆文件来执行，而不是通过独立后端驱动。

它对外应表现为一个会自己判断下一步的协作助手，而不是一个要求 PM 操作状态机的流程系统。

## 2. 固定动作流

对外固定动作只保留九类：

- 听需求
- 先回应
- 对齐
- 开工检查
- 做原型
- 写 PRD
- 评审
- 改需求
- 修问题

实现映射如下：

- 听需求 -> `omp-listen`
- 先回应 -> `omp-reply`
- 对齐 -> `omp-align`
- 开工检查 -> `omp-ready`
- 做原型 -> `omp-proto`
- 写 PRD -> `omp-prd`
- 评审 -> `omp-review`
- 改需求 -> `omp-change`
- 修问题 -> `omp-fix`

补充的通用实现包括：

- `omp-check`
- `omp-memory`
- `omp-governance`

## 3. 五层加载

运行时固定按以下五层推进：

1. 入口层：判断当前意图、主控权和真实/样例场景
2. 状态层：只读最小状态与最小项目记忆摘要
3. 决策层：只读当前动作对应的单个 skill 与必要规则
4. 交付层：仅在重动作时读取交付规则、局部材料和稳定基线
5. 归档层：回写稳定路径、摘要、索引和状态

默认禁止：

- 一上来读取多个 skill
- 一上来读取很多规则
- 一上来载入整篇长材料和整包历史

## 3.1 单一主控入口

运行时主控入口固定为：

```powershell
powershell -File .\scripts\control\ompgo.ps1
```

它是所有自然语言协作与强制入口共用的总装层。

固定执行链：

1. `scripts/tools/state-machine.ps1`：读取最小状态，判断当前主链位置
2. `scripts/tools/route-resolve.ps1`：把自然语言或强制入口映射成单个动作
3. `scripts/control/ompgo.ps1`：汇总门禁、ask-back 和唯一收口
4. 决策层：只装当前动作对应的一个 skill 与必要规则
5. 交付层：仅在重动作时补读交付材料
6. 归档层：回写状态并只保留唯一下一步

## 4. 主链

固定主链为：

`听需求 -> 先回应 -> 对齐(循环) -> 开工检查 -> 做原型 / 写 PRD -> 评审 -> 改需求`

说明：

- `先回应` 与 `对齐` 构成主循环
- `做原型` 与 `写 PRD` 都属于正式交付动作
- `评审` 负责给出统一结论，而不是普通确认
- `改需求` 只处理正式交付后的新增或修正，不直接污染当前稳定方案

## 5. 动作说明

### 5.1 听需求

目标：

- 保留业务原话
- 建立当前需求任务
- 判断是否具备进入先回应的最小输入

### 5.2 先回应

目标：

- 给出当前理解
- 给出当前版本方案草案
- 暴露不确定项和待确认项

阻断时：

- 若关键事实缺失，优先转为 ask-back

### 5.3 对齐

目标：

- 根据反馈修正理解
- 更新方案、变化点、模块判断和粗估
- 判断是否对齐，还是进入开工检查

说明：

- 需求输入可以继续补充
- 系统只锁推进动作，不锁输入

### 5.4 开工检查

目标：

- 判断当前方案是否已经足够稳定，可以进入正式交付

固定检查项：

- 范围闭合
- 流程闭合
- 模块闭合
- 未澄清项风险
- 工时是否可解释
- 交付承接是否完整

### 5.5 做原型

目标：

- 生成交付型原型
- 让研发和评审先通过页面、流程和标注理解方案

### 5.6 写 PRD

目标：

- 生成正式归档主文件
- 补足规则、异常、权限、数据影响和验收说明

### 5.7 评审

目标：

- 用多角色评审团对当前版本做统一评审
- 输出通过、条件通过、打回或暂缓结论

### 5.8 改需求

目标：

- 接收正式交付后的新增诉求或修正诉求
- 判断它是并入、修复、重开对齐还是正式变更

### 5.9 修问题

目标：

- 修正评审或交付过程中发现的问题
- 必要时触发下游修正上游

## 6. ask-back 在流程中的位置

ask-back 不是独立主流程，而是所有阶段共享的阻塞解除器。

规则：

- 当当前动作因信息不足而无法继续时，优先进入 ask-back
- ask-back 每次只问一个最阻塞的问题
- ask-back 只能服务真实协作阻塞，样例业务口径不得抛给 PM

## 7. 宿主运行顺序

宿主执行顺序固定为：

1. 判断入口
2. 读取最小状态
3. 读取当前动作对应的 skill
4. 读取当前动作必需规则
5. 必要时进行上下文控制
6. 必要时提唯一问题
7. 执行当前动作
8. 必要时执行评审或复写
9. 回写状态和记忆

脚本层固定分为：

- `scripts/control/`：主控入口、初始化、链路回放
- `scripts/tools/`：门禁、ask-back、状态回写、记忆回写、评审和复写等底层工具

## 8. 与 ShitPM 的差异

ShitPM 更强调正式主线推进与正式产物控制。  
OhMyPm 当前更强调：

- 以自然语言协作入口为主
- 以小上下文逐层加载为主
- 以回应与对齐循环为主
- 以项目记忆和系统记忆作为长期主存
