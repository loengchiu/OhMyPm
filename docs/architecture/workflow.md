# OhMyPm Skill 工作流

## 1. 运行形态

OhMyPm 运行在 AI IDE / AI CLI 中。  
宿主模型通过读取项目目录内的规则、状态和记忆文件来执行 skill，而不是通过独立后端驱动。

主流程已定，不再围绕“阶段产物链”组织，而是围绕 PM 的真实需求处理流程组织。

## 2. Skill 分层

### 2.1 阶段型 skill

阶段型 skill 负责推进主流程中的明确节点。

建议首批包含：

- `omp-intake`：需求接收
- `omp-respond`：形成当前理解与一版可信回应
- `omp-align`：回应/校验循环中的修正推进
- `omp-preflight`：正式交付前检查
- `omp-deliver-prototype`：交付型原型生成
- `omp-deliver-prd`：PRD 生成
- `omp-review`：评审会
- `omp-change`：正式交付后的变更处理
- `omp-fix`：针对评审或产物缺陷的定点修复

### 2.2 通用型 skill

通用型 skill 不直接代表一个业务阶段，但可以被多个阶段复用。

建议首批包含：

- `omp-gate-check`：执行门禁判断
- `omp-ask-back`：执行追问
- `omp-context-guard`：执行上下文规模评估与分块
- `omp-memory-read`：读取项目记忆和系统记忆
- `omp-memory-write`：回写项目记忆和系统记忆
- `omp-review-panel`：执行评审机制内的多角色评审团
- `omp-overwrite-judge`：执行下游修正上游判定
- `omp-estimate`：执行模块级粗工时判断
- `omp-artifact-sync`：同步状态、基线和产物索引

## 3. 激活规则

### 3.1 激活开关

建议以 `docs/ohmypm/ohmypm-status.json` 作为 OhMyPm 在当前项目中的激活开关。

规则：

- 文件不存在：视为未初始化，只允许执行初始化或纯咨询
- 文件存在：OhMyPm 进入正式工作流模式

### 3.2 宿主最小行为约束

当 OhMyPm 已激活时，宿主模型在执行阶段型 skill 之前，必须至少完成以下动作：

1. 读取 `docs/ohmypm/ohmypm-status.json`
2. 读取 `docs/ohmypm/ohmypm-memory.md`
3. 根据任务需要读取相关 `docs/ohmypm/system-memory/*.md`
4. 执行门禁判断
5. 执行上下文风险判断

如果以上动作未完成，不得表述为“已确认可推进”。

## 4. 主流程

OhMyPm 的主流程已定为：

- `需求接收`
- `回应/校验循环`
- `交付前检查`
- `正式交付`
- `变更控制`

映射到 skill 主链可表示为：

`omp-intake -> omp-respond -> omp-align(loop) -> omp-preflight -> omp-deliver-prototype / omp-deliver-prd -> omp-review -> omp-change`

说明：

- `omp-respond` 与 `omp-align` 共同构成核心循环
- `omp-deliver-prototype` 与 `omp-deliver-prd` 可根据交付策略并行或先后执行
- `omp-review` 可以位于正式交付前后的关键节点，但必须带结论
- `omp-change` 负责处理正式交付后的新增或修正，不允许直接污染当前稳定方案

对应规则文件：

- 门禁与回退：`contracts/gates.md`
- 追问：`contracts/ask-back.md`
- 记忆：`contracts/memory.md`
- 上下文控制：`contracts/context-guard.md`
- 正式交付边界：`contracts/delivery.md`
- 评审：`contracts/review.md`
- 复写：`contracts/overwrite.md`

## 5. 阶段说明

### 5.1 `omp-intake`

目标：

- 保存业务原话
- 建立当前需求任务
- 判断是否具备进入第一轮回应的最小输入

产物：

- 初始状态文件
- 项目记忆文件初稿

门禁：

- 若需求任务都无法定义，不能进入 `omp-respond`

### 5.2 `omp-respond`

目标：

- 给出当前理解
- 给出当前版本方案草案
- 暴露不确定项和追问点

产物：

- 回应摘要
- 当前版本方案
- 待确认项

门禁：

- 若关键事实缺失，必须先触发 `omp-ask-back`

### 5.3 `omp-align`

目标：

- 根据需求方反馈修正理解
- 更新方案、变化点、模块判断和粗估
- 判断是否继续下一轮，或进入 `omp-preflight`

说明：

- 需求方可以持续补材料、补需求
- 系统不锁需求输入，但锁推进动作

产物：

- 新版本方案
- 本轮变化点
- 模块与工时判断

门禁：

- 主结构仍不稳定时，不得进入正式交付

### 5.4 `omp-preflight`

目标：

- 执行交付前检查
- 判断目标、主流程、模块、未澄清项和表达路径是否闭合

产物：

- 交付前检查结论
- 阻塞项或放行结论

门禁：

- 六项闭合条件未满足时，禁止进入正式交付

### 5.5 `omp-deliver-prototype`

目标：

- 生成交付型原型
- 明确页面落点、主流程、页面用途、用户动作、关键状态、关键交互标注和页面间关系

门禁：

- 对齐条件和交付前检查未通过时，不得执行

### 5.6 `omp-deliver-prd`

目标：

- 生成正式归档所需 PRD
- 补足业务规则、异常、权限、数据影响和验收说明

规则：

- PRD 是正式归档主文件
- 在评审和研发理解中，原型为主、PRD 为补充
- 在正式归档中，PRD 为主文件、原型为核心可视化附件

门禁：

- 原型与 PRD 之间的表达边界必须先判清
- 若内容过大，必须启用分块生成

### 5.7 `omp-review`

目标：

- 用多角色评审团对当前可评审版本进行审查
- 产出通过、条件通过、打回或暂缓结论

门禁：

- 没有可评审版本时不得进入

### 5.8 `omp-change`

目标：

- 接收正式交付后的新增诉求或修正诉求
- 判断它是小修、补充还是范围变更
- 决定进入修复、复写还是新一轮对齐

门禁：

- 不得把新增内容直接混入当前稳定版本

## 6. 追问机制在流程中的位置

追问不是单独阶段，而是所有阶段共享的阻塞解除器。

规则：

- 当门禁失败原因是“信息不足”时，优先触发 `omp-ask-back`
- 追问结果先回写项目记忆，再继续原阶段
- 同一轮优先只追最影响推进的一个问题

## 7. 多角色评审团在流程中的位置

多角色评审团不是独立主机制，而是评审机制内部的默认审查方式。

默认触发点：

- `omp-preflight`：做小范围交付前审查
- `omp-review`：做正式评审
- `omp-change`：判断新增需求是否会破坏原有边界

输出要求：

- 汇总事实问题
- 汇总风险问题
- 汇总建议问题
- 给出统一结论和下一步动作

## 8. 下游修正上游机制在流程中的位置

默认触发点：

- `omp-deliver-prd` 发现上游方案缺口
- `omp-deliver-prototype` 发现页面结构与方案冲突
- `omp-review` 给出推翻性结论
- `omp-fix` 证明稳定基线失效

触发后流程：

1. 进入 `omp-overwrite-judge`
2. 判定影响范围
3. 选择局部修正、整体回退或重开对齐
4. 回写上游文件、状态文件、项目记忆

## 9. 宿主运行规则

为了保证 AI IDE / CLI 中的 skill 可稳定运行，宿主应遵循以下顺序：

1. 读状态
2. 读项目记忆
3. 按需读取外部知识主仓中的相关资料，并读取已存在的系统记忆卡
4. 执行门禁
5. 执行当前阶段内置的方法论规则
6. 执行上下文风险判断
7. 必要时追问
8. 执行当前阶段 skill
9. 必要时执行多角色评审团
10. 必要时执行复写判定
11. 回写状态和记忆

## 10. 与 ShitPM 的差异

ShitPM 更强调线性阶段推进与文档产物控制。  
OhMyPm 需要保留这些优点，但把主流程重心改成：

- 以回应/校验循环为中心
- 以项目记忆和系统记忆为长期主存
- 以追问和评审机制作为认知控制器
- 以复写机制保障前后链路一致性
