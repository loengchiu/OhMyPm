# 第一版交付型原型草稿

## 1. 原型范围
- 本稿对应当前版本内的审批路径扩展。
- 本稿只覆盖现有审批能力内部增强，不单独拆出新模块。
- 本稿用于评审理解和研发第一阅读，不替代后续 PRD 规则正文。

## 2. 这版原型至少覆盖的页面

### P1 审批入口与待办列表页
页面用途：
- 展示审批能力主入口。
- 展示当前用户可进入的待办、已办或相关任务列表。
- 标出新增审批路径从哪里进入。

本页至少要标的点：
- 入口位置
- 新增路径触发条件
- 列表状态标签变化
- 哪些状态变化会影响后续查询结果

### P2 审批详情与处理页
页面用途：
- 展示单据或审批对象的核心信息。
- 展示审批节点、当前状态、可执行动作。
- 作为新增审批路径主流程的核心页面。

本页至少要标的点：
- 当前处于哪个审批节点
- 当前可以执行什么动作
- 新增审批分支在什么条件下出现
- 哪些动作会改变状态、通知或记录

### P3 审批流转记录 / 轨迹视图
页面用途：
- 展示审批过程中的关键节点变化和处理记录。
- 承接留痕与追溯要求。

本页至少要标的点：
- 关键节点进入时间
- 处理动作结果
- 驳回、退回、重新提交等异常记录
- 新增路径是否进入过特殊分支

### P4 通知触达视图
页面用途：
- 展示哪些审批动作会触发通知。
- 展示通知是否沿用原规则，还是因新增路径需要补充例外场景。

本页至少要标的点：
- 哪些节点触发通知
- 通知对象是谁
- 哪些场景不触发通知
- 通知结果与状态结果是否一致

## 3. 这版原型至少覆盖的流程

### F1 主流程
最少覆盖：
- 从审批入口进入
- 进入审批详情
- 经过新增审批路径主链路
- 完成审批并回到结果状态

原型上要看清：
- 主链路怎么走
- 哪些节点是这次新增影响点
- 结束后状态如何呈现

### F2 关键分支流程
最少覆盖：
- 原有路径继续沿用的分支
- 新增审批路径触发的分支
- 会改变结果判断的条件分支

原型上要看清：
- 哪些分支只是规则变化
- 哪些分支会影响状态、通知或记录

### F3 异常流程
最少覆盖：
- 驳回
- 退回
- 重新提交
- 流程中断

原型上要看清：
- 异常从哪里发生
- 异常后回到哪里
- 异常记录怎么保留

### F4 联动流程
最少覆盖：
- 状态变化后列表结果是否变化
- 状态变化后通知是否触发
- 状态变化后记录是否更新

原型上要看清：
- 同一个动作对页面、通知、记录分别带来什么影响

## 4. 原型编号标注建议
- `01` 入口位置与触发条件
- `02` 新增审批路径进入条件
- `03` 主流程关键节点
- `04` 结果判断分支
- `05` 状态变化点
- `06` 通知触发点
- `07` 留痕记录点
- `08` 驳回 / 退回 / 重新提交位置
- `09` 需要在 PRD 中补正式规则的位置

## 5. 原型与 PRD 分工提醒
- 原型负责把页面、流程、状态和关键变化点讲清。
- PRD 负责把判定条件、通知规则、留痕规则、异常规则和验收标准写完整。
- 原型中不铺大段正文，只通过编号把需要解释的位置挂出来。

## 6. 当前结论
- 这版原型的最小覆盖范围已经明确。
- 如果继续往下做，下一步就不是再补说明，而是按这四类页面和四类流程展开第一版原型结构。
