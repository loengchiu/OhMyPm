# 交付型原型承接稿

## 1. 本稿的作用
- 本稿用于把当前稳定对齐包、接口正式承接稿、验收正式承接稿接到交付型原型。
- 原型阶段不再重新讨论范围和模块归属，只负责把已经确认的方案通过页面和流程展示出来。

## 2. 原型展示目标
- 让评审会能一眼看懂这次审批路径扩展改了什么。
- 让研发先通过页面关系、流程走向、关键状态理解方案主干。
- 让后续 PRD 可以专注写规则、异常、权限、数据影响和验收，而不是重复解释主流程。

## 3. 原型展示边界
- 本轮原型仍以审批能力为主入口，不单独包装成一个新模块。
- 原型重点展示：
  - 新增审批路径落在哪些页面或页面区域
  - 主流程怎么进入、流转、结束
  - 关键分支在哪里发生变化
  - 关键状态如何变化
  - 哪些地方需要回看 PRD 规则说明

## 4. 原型最少应包含的页面与视角

### 4.1 主入口视角
- 展示审批能力的主入口位置。
- 标出本轮新增审批路径从哪里进入。

### 4.2 主流程视角
- 展示新增审批路径的主链路。
- 标出关键节点之间的页面关系和跳转关系。

### 4.3 分支流程视角
- 展示会改变结果判断的关键审批分支。
- 标出哪些分支只是规则调整，哪些分支会引起状态、通知或记录变化。

### 4.4 状态与结果视角
- 展示关键状态在流程中的变化点。
- 标出哪些状态会影响查询结果、消息提醒、记录留痕或统计口径。

### 4.5 异常处理视角
- 展示驳回、退回、重新提交、流程中断等异常处理位置。
- 标出哪些异常仍沿用原规则，哪些异常需要在 PRD 中看正式说明。

## 5. 原型标注重点
- 对关键节点使用编号标注，不在页面铺大段文字。
- 编号解释重点写：
  - 这个节点在做什么
  - 这里为什么和原流程不同
  - 会影响什么状态、通知或记录
  - 是否需要引用 PRD 中的正式规则

## 6. 原型与 PRD 的分工
- 原型负责：
  - 页面落点
  - 主流程
  - 关键分支
  - 关键状态变化
  - 页面关系
- PRD 负责：
  - 判定条件
  - 状态同步规则
  - 通知联动规则
  - 留痕规则
  - 异常与回退规则
  - 验收标准

## 7. 当前原型承接结论
- 当前已经具备第一版原型承接基础。
- 原型现在可以开始围绕审批能力主入口、审批路径主链路、关键分支和关键状态做第一版展示设计。
- 但在正式交付门禁未通过前，这份稿仍属于原型准备基础，而不是正式交付成品。
