# 后续材料继承说明

## 1. 这份说明的作用
- 这份说明用于规定：后续原型、PRD、评审材料如何直接继承当前稳定对齐包，而不是各自重新判断范围、模块和边界。

## 2. 统一继承原则
- 后续材料默认继承 `docs/ohmypm/alignment/support/stable-alignment-package.md`。
- 若后续材料与稳定对齐包冲突，应优先回到对齐层修正，不得各自写出不同版本。
- 后续材料可以展开细节，但不应改写已经确认过的范围和模块归属。

## 3. 原型应继承什么
- 继承当前范围结论：新增审批路径扩展属于现有审批能力内部增强。
- 继承当前模块拆分：原型中应继续以审批能力为主入口展开，而不是额外包装成独立新模块。
- 继承当前主流程和关键分支：原型需要把新增审批路径、关键状态、关键交互、页面间关系标出来。
- 原型重点负责“怎么走、在哪做、状态怎么变化”，不负责重复大段规则正文。

## 4. PRD 应继承什么
- 继承当前接口边界口径：把状态同步、通知联动、记录留痕、异常回退等内容写成正式规则说明。
- 继承当前验收覆盖口径：把主流程、分支流程、异常流程、联动影响、兼容性要求转成正式验收描述。
- 继承当前工时与排期口径：PRD 中可以说明影响来源，但不应把本轮内容改写成独立新模块。
- PRD 重点负责“规则、条件、异常、权限、数据影响、验收”，不负责重新解释页面主流程。

## 5. 评审材料应继承什么
- 继承当前范围与模块归属结论，避免评审会上重新争论已确认事项。
- 继承当前工时与排期表达：明确是原模块内复杂度上升，不是独立新模块新增。
- 继承当前接口边界和验收覆盖缺口：把它们作为评审关注点，而不是让评审从零猜测哪里还没补齐。

## 6. 当前使用顺序
- 先用 `docs/ohmypm/alignment/support/stable-alignment-package.md` 作为统一说明。
- 再按需要分别展开到原型、PRD、评审材料。
- 若后续出现新反馈，应先修正稳定对齐包，再同步带动下游材料。
