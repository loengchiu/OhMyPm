# 接口与边界正式承接稿

## 1. 适用范围
- 本稿适用于当前版本内的审批路径扩展。
- 本稿用于正式承接稳定对齐包中的接口边界结论，作为后续 PRD 中“业务规则说明”“异常与边界说明”“角色、权限与数据影响”相关内容的直接基础。

## 2. 总体边界结论
- 本轮内容仍属于现有审批能力内部增强，不形成新的独立外部能力边界。
- 本轮重点是补充原审批能力下的路径分支、状态同步、通知联动、记录留痕和异常处理规则。
- 对外表达时，应统一使用“沿用原能力框架，在原边界内补充新增规则”的口径。

## 3. 审批路径边界
- 需要明确本轮新增审批分支的进入条件、流转节点和结束条件。
- 需要区分“仅规则调整”的分支与“会带来展示、状态、通知变化”的分支。
- 对于仅规则调整的分支，可沿用现有流程框架表达；对于会影响联动结果的分支，必须补写对应规则说明。

## 4. 状态同步规则
- 审批路径扩展后，凡是会影响页面显示、查询结果、消息提醒、统计口径的状态变化，都必须有明确同步规则。
- 状态同步至少应说明：
  - 什么时候变
  - 变成什么
  - 影响哪些下游位置
  - 是否需要补充例外说明

## 5. 通知联动规则
- 新增审批路径涉及通知时，必须明确是沿用现有通知规则，还是新增例外场景。
- 若沿用现有通知规则，应说明适用条件。
- 若存在例外场景，应说明触发条件、通知对象和不触发条件。

## 6. 记录留痕规则
- 凡是会影响审批责任、状态判断或追溯结论的动作，必须保留记录。
- 记录留痕至少应覆盖：
  - 关键节点进入
  - 处理动作执行
  - 驳回、退回、重新提交等异常动作
  - 会影响结果判断的状态切换

## 7. 异常与回退规则
- 本轮必须明确哪些异常继续沿用原规则，哪些异常需要补充新规则。
- 异常说明至少应覆盖：
  - 驳回
  - 退回
  - 重新提交
  - 流程中断
  - 关键联动失败后的处理口径

## 8. 正式材料使用要求
- 原型负责把关键状态、关键分支和页面关系标出来。
- PRD 负责把本稿中的同步规则、通知规则、留痕规则和异常规则写成正式可归档内容。
- 若后续正式材料与本稿冲突，应先回到对齐层修订，不得各自保留不同版本。
