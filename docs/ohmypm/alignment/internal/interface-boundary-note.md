# 接口边界说明

## 1. 本说明解决什么问题
- 这份说明用于回答“本轮审批路径扩展会影响哪些系统边界、哪些联动口径、哪些地方需要在正式交付前讲清楚”。
- 重点不是先写技术接口细节，而是先把产品侧必须说清楚的边界补齐。

## 2. 当前边界结论
- 本轮新增内容仍落在现有审批能力内部。
- 当前没有形成新的独立外部能力边界。
- 但审批路径扩展会放大原审批能力与通知、记录、状态同步等既有联动关系，因此需要把这些联动边界明确写出来。

## 3. 当前需要说清的边界

### 3.1 审批路径定义边界
- 需要明确哪些审批分支属于本轮新增范围。
- 需要明确哪些分支只是规则调整，哪些分支会引起页面展示、流转记录或状态判断变化。

### 3.2 状态同步边界
- 需要明确审批路径扩展后，哪些关键状态会变化。
- 需要明确状态变化是否会影响下游页面显示、查询结果、消息提醒或统计口径。

### 3.3 通知联动边界
- 需要明确新增审批路径是否触发新的通知场景，还是沿用现有通知规则。
- 若沿用现有规则，需要明确哪些条件下复用，哪些条件下需要补充例外说明。

### 3.4 记录留痕边界
- 需要明确新增审批路径扩展后，哪些动作必须留痕。
- 需要明确留痕粒度是否与现有审批链路一致，还是需要新增关键记录点。

### 3.5 异常回退边界
- 需要明确审批路径扩展后，异常中断、驳回、退回、重新提交等情况是否改变原有处理方式。
- 需要明确哪些异常仍按原规则处理，哪些异常需要单独补充说明。

## 4. 当前建议口径
- 对外说明时，应优先使用“沿用哪些原规则、补充哪些新增边界”这一表达方式。
- 不要把它说成“新增了一整套接口体系”，否则容易误导为独立新模块或独立系统改造。

## 5. 进入更重阶段前至少要补齐的内容
- 一张审批路径扩展影响点清单
- 一组状态同步规则说明
- 一组通知联动规则说明
- 一组关键留痕与异常处理说明
