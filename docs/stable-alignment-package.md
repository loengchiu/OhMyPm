# 稳定对齐包

## 1. 当前结论
- 本轮审批路径扩展仍属于当前版本范围。
- 本轮新增内容仍属于现有审批能力内部增强，不单独拆成新模块。
- 当前可以继续做对齐推进，但还不进入更重交付动作。

## 2. 当前版本方案
- 继续按当前版本推进审批能力增强。
- 把新增审批路径扩展并入现有审批能力说明，不单独拆出新的独立交付边界。
- 对外说明时统一采用“模块不变、复杂度上升、因此工时和排期受影响”的表达口径。

## 3. 当前模块拆分
- 审批能力
  - 既有审批流程
  - 本轮新增的审批路径扩展
- 通知模块

补充说明：
- 当前没有新增独立页面组。
- 当前没有形成需要单独命名、单独交付的独立能力块。

## 4. 当前工时与排期口径
- 本轮工作量增加，来源是现有审批能力内部复杂度上升。
- 增量主要来自审批分支梳理、规则说明、边界处理、联动验证。
- 当前排期影响应表述为：仍在原模块内消化，但由于复杂度上升，需要上调工时和排期判断。
- 不应把工作量增加直接说成“新增了一个模块”。

## 5. 当前接口边界口径
- 本轮重点不是新增独立接口体系，而是补清原审批能力下的联动边界。
- 当前至少要说清以下边界：
  - 哪些审批分支属于本轮新增范围
  - 哪些状态变化会影响下游页面、查询结果、消息提醒或统计口径
  - 哪些通知规则沿用原规则，哪些场景需要例外说明
  - 哪些关键动作必须留痕
  - 哪些异常仍按原规则处理，哪些异常需要新增说明

## 6. 当前验收覆盖口径
- 验收不能只证明“新增路径可用”，还必须证明“原路径未被破坏”。
- 当前至少应覆盖：
  - 主流程验收
  - 分支流程验收
  - 异常流程验收
  - 通知、记录、查询结果等联动影响验收
  - 兼容性验收

## 7. 当前还不能进入更重阶段的原因
- 模块范围和模块归属已经定了，不再是当前阻塞点。
- 当前真正剩下的是把上面几部分表达统一成后续可直接复用的正式说明。
- 在这套统一说明没有被后续材料承接前，还不进入正式原型或正式 PRD。

## 8. 后续衔接建议
- 后续继续推进时，优先以这份稳定对齐包为主说明。
- 原型、PRD、评审材料应在此基础上展开，不再各自重新解释范围和模块归属。
