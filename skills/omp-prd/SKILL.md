# 写 PRD

## 最小读取

- 先读 `.ohmypm/status.json`
- 再读 `.ohmypm/memory.md` 的最小必要摘要
- 读取 `context_risk`；若不存在，先按最小结构补齐
- 当前只执行 `omp-prd`
- 默认只补 `contracts/delivery.md`、`contracts/anchors.md`、`contracts/anchors-and-trace.md`
- 门禁异常补 `contracts/gates.md`；命中长文、跨模块或跨产物联动时补 `contracts/context-guard.md`
- 系统记忆和外部知识只允许局部回查
- 只有在判断章节展开粒度、动作写法或文风取舍时，才补读 `reference/prd-writing.md`

## 目标

- 生成正式归档主文件 PRD
- 按 OMP 正式归档骨架组织内容，并显式补齐数据字典章节
- 保证 PRD 规则正文挂在当前模块 / 页面 / 动作锚点上
- 根据 `solution_shape` 选择写法重点
- 章节目录可按 OMP 骨架组织，正文写法、文风和规格表达默认对齐当前已验证成熟的 PRD 写法
- PRD 本身应支持独立归档、独立评审、独立实施；原型用于辅助理解页面呈现，PRD 负责完整交代规则、边界、权限、数据字典、数据影响和验收

## 必读状态

- `baselines.response_plan`
- `baselines.prototype`
- `context_package.solution_shape`
- `alignment_state.round_result`
- `fallback_state`
- `anchors_state.meta.anchor_manifest`
- `baselines.prd`
- `solution-shape diff`

## 执行顺序

1. 检查正式交付门禁
2. 读取原型与交付规则边界
3. 若需要系统记忆或外部知识，只允许读摘要、索引和局部片段
4. 分块写 PRD
5. 按补漏清单检查 PRD 缺口
6. 汇总长文摘要并回收上下文
7. 回写 PRD 基线与产物路径
8. 执行 `trace-check`

详细需求说明阶段固定附加规则：

1. 先按模块切块
2. 再按页面切块
3. 最后按动作切块
4. 每个动作都必须有对应锚点

## 最低输出

- 一版按正式归档骨架组织的 PRD
- 与原型互补而不重复的规则说明
- 异常、边界、权限、数据字典、数据影响和验收说明
- 一份补漏结果摘要

## 强制规则

- 未通过正式交付门禁，不得开始
- 长文必须分块生成并做摘要回收
- PRD 是否需要分块，不得只按字数判断；必须同时看模块范围、页面范围、是否联动原型和规则复杂度
- `context_risk` 的 `length_signals / complexity_signals / decision` 必须使用 `contracts/context-guard.md` 中的标准口径
- 不得为了完整感擅自新增模块、页面、流程、规则、字段、验收项
- 只能补“当前范围内必需内容”；其余标为待确认或不写
- 模块、页面、动作的新增、删除或范围扩张必须由 PM 确认，不得自行扩充
- 详细需求说明必须遵守模块、页面、动作三层锚点规则
- PRD 不得脱离当前追溯元数据和锚点扩写
- PRD 生成时必须读取内部 `anchor_manifest`；正文标题使用自然标题，页面最多保留 `P1 / P2` 这类页面编号，完整组合锚点只保存在内部 manifest
- PRD 生成后必须执行 `python scripts/python/omp-lint.py trace-check --status-path .ohmypm/status.json`；结果为 `fail` 时不得进入评审，必须先修正 manifest、路径或机读字段泄漏
- PRD 只写正式归档正文，不写模板说明、方法论说明、元话语
- PRD 不出现 `AI`、绝对路径、调试路径、试跑结论等内部表述
- PRD 骨架用于兜底范围，不用于逐条填空；每章按内容复杂度选择自然段、短列表或表格，不机械重复模板标签
- 正文使用自然标题、短段落、短列表和必要表格；不使用 `**内容标签**：` 撑结构，不把一句话拆成很多小标题
- 简单章节短写，复杂章节细写；详细需求说明按“模块 → 页面 → 动作”展开
- 简单动作 1-2 句直接写清；中等复杂动作补顺序和状态变化；高复杂动作写清触发、顺序、状态、分支和中断
- 静态文案、动态来源、数值阈值、超长文本处理、错误提示、边界限制必须写死，不用“按配置”“详见实现”这类占位说法
- 对外资料引用使用人类可读名称，路径、链路和调试信息保留在内部状态或证据文件
- 动作级覆盖清单不是可选装饰。涉及对应场景时，必须显式补齐：
  - 数据展示：静态文案、动态来源、无数据、超长文本处理
  - 按钮与操作：可点击/不可点击条件、点击反馈、成功变化、失败提示
  - 表单与输入：输入限制、校验时机、错误提示文案
  - 列表与加载：分页或加载方式、加载中、空状态、失败状态
  - 弹窗与抽屉：触发方式、关闭方式、遮罩行为、多弹层优先级
  - 异常与降级：网络异常、权限不足、关键功能失效时的降级处理
  - 边界：数值边界、并发冲突、状态临界值、导入导出或统计口径限制
- PRD 必须读取“建设类型差异说明表”，按其中的 PRD 重点补足规则，不得自行切换为另一套结构
- 只要需求涉及新增数据存储、展示字段、筛选字段、统计口径、接口映射或导出字段，PRD 必须显式给出“数据字典与数据影响”章节，不得只在正文散写字段名称
- 详细需求说明中的字段信息默认按“字段分组 + 关键例外”表达，不把完整数据字典逐页搬进正文
- 字段分组按需出现：只有字段较多、存在明显业务分块、或字段归属会影响理解与实现时才写；简单页、轻列表、纯结果页可以不写
- 字段分组只服务于页面理解，不得为了模板齐整补空泛分组；没有分组价值时宁可不写
- 页面正文中如出现字段规则，只写当前页面真正影响理解、实现或评审的关键例外，不机械逐字段展开
- 全局“数据字典与数据影响”章节继续保留完整字段汇总；详细需求说明是字段主编辑入口，数据字典是必须同步的汇总层
- 从 `omp-proto` 切换进入 `omp-prd` 时，必须把“读状态 → 读 `omp-prd` skill → 执行最小读取 → 检查正式交付门禁”作为完整新动作执行；不得把 `/prd` 仅理解成继续沿用原型上下文补文档
- 当 `solution_shape=iteration` 时，PRD 必须显式写清改造说明、挂载点、兼容约束和存量影响
- 当 `solution_shape=new_build` 时，PRD 必须显式写清新建范围、模块结构、角色边界和主流程规则
- 当 `solution_shape=hybrid` 时，PRD 必须把存量承接规则和新建规则分段写清
- PRD 初稿生成后必须执行一轮补漏检查，至少覆盖字段落地、字段一致性、权限一致性、状态一致性、变更传播和动作级场景覆盖
- 若当前原型尚未稳定，PRD 不得伪装为最终归档版本
- PRD 生成完成后默认停在当前阶段；只有 PM 显式执行 `/review`，才允许进入评审动作

  - `baselines.prd`

  - `context_risk.decision`
  - `context_risk.last_updated`
